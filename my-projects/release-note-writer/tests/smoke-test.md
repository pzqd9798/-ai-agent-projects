# Smoke Test: release-note-writer

## 目的

验证 `release-note-writer` 技能是否**确实提升了任务成功率**。

## 测试前提

- 当前目录是一个 git 仓库
- 历史中有至少 5 个提交
- 提交 message 尽量遵循 Conventional Commits 规范

## 对照组设计

| 条件 | 方法 | 预期 |
|------|------|------|
| **无技能（基线）** | 直接对 Claude 说"写 Release Note" | Claude 会猜测项目内容，从已有上下文拼凑内容，可能遗漏实际提交 |
| **有技能** | 调用 `/release-note-writer` | Claude 执行 `git log` 收集实际提交、按约定分类、填充模板、运行验证脚本 |

## 测试步骤

### 1. 准备测试环境

```bash
# 进入一个有提交历史的项目
cd /path/to/your-project

# 确认有足够提交
git log --oneline --no-merges | wc -l
# 输出应 ≥ 5
```

### 2. 基线测试（无技能）

在 Claude Code 中输入（**确保技能目录不存在**或临时移走）：

```
帮我写一份这个版本的发版说明 / Release Notes
```

记录：
- 输出是否引用了真实的 git 提交？ 是 ☐ 否 ☐
- 是否正确分类（features/fixes）？ 是 ☐ 否 ☐ 部分 ☐
- 是否有遗漏的重要提交？ 是 ☐ 否 ☐
- 格式是否统一规范？ 是 ☐ 否 ☐
- **从撰写到满意共修改了几次？** ____ 次

### 3. 有技能测试

在 Claude Code 中输入：

```
/release-note-writer v1.2.0
```

或：

```
/release-note-writer
```

记录：
- 输出是否引用了真实的 git 提交？ 是 ☐ 否 ☐
- 是否正确分类（features/fixes）？ 是 ☐ 否 ☐ 部分 ☐
- 是否有遗漏的重要提交？ 是 ☐ 否 ☐
- 格式是否统一规范？ 是 ☐ 否 ☐
- 验证脚本是否通过？ 是 ☐ 否 ☐ 未运行 ☐
- **从撰写到满意共修改了几次？** ____ 次

### 4. 对比分析

| 指标 | 无技能（基线） | 有技能 | 改善 |
|------|:------------:|:-----:|:----:|
| 引用真实 git 提交 | | | |
| 正确分类 | | | |
| 无重要遗漏 | | | |
| 格式统一 | | | |
| 修改次数 | | | |

### 5. 边缘情况测试

| 场景 | 命令 | 预期行为 |
|------|------|----------|
| 无 tag 历史 | `/release-note-writer` | 使用最近 N 个提交 |
| 只有一个提交 | `/release-note-writer` | 仍能产出单条 Release Note |
| 非 git 目录 | `/release-note-writer` | 友好报错，告知需要 git 仓库 |
| 指定起始 tag | `/release-note-writer --since=v1.0.0` | 从 v1.0.0 到 HEAD |
| 无新提交 | `/release-note-writer` | 提示"无新变更" |

## 成功标准

如果**有技能**比**无技能**满足以下条件，则 smoke test 通过：

1. **修改次数降低**：有技能 ≤ 1 次修改即满意，基线通常需 2+ 次
2. **提交引用准确率提升**：有技能 100% 引用真实提交，基线常遗漏
3. **格式一致性**：有技能产出格式稳定，基线格式随意
4. **验证脚本通过**：`verify-checklist.sh` 一次通过

## 结论

- [ ] ✅ 通过 — 技能确实提升任务成功率
- [ ] ⚠️ 部分通过 — 格式改善但内容准确性无显著变化
- [ ] ❌ 未通过 — 技能未带来可观测的改善
